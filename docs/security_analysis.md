#  Spoon - Backend Security & Config Deep Dive Analysis
> **작성자**: 10년 차 백엔드 아키텍트 (Antigravity)
> **대상 패키지**: `com.project.hanspoon.common.config`

이 문서는 한스푼 프로젝트의 핵심 보안 및 가동 설정 파일들을 해부하여 분석한 내용을 담고 있습니다.

---

## 🛡️ 1. SecurityConfig.java (성벽의 설계도)

이 파일은 우리 애플리케이션의 가장 최전방 방어선입니다. 어떤 길(Path)은 열어주고, 어떤 길은 보초(Auth)를 세울지 결정합니다.

### 주요 해부점
- **CORS 설정**: 서로 다른 도메인(프론트-백) 간의 통신을 허용합니다.
- **CSRF Disable**: 세션을 사용하지 않고 JWT 토큰을 사용하여 CSRF 보호를 끕니다. (성능 최적화)
- **STATELESS**: 서버는 사용자의 정보를 메모리에 저장하지 않습니다. 오직 '토큰'만 믿고 판단합니다. (확장성 확보)
- **최소 권한 원칙**: "명시하지 않은 모든 길은 검문한다" (`anyRequest().authenticated()`)를 충실히 따릅니다.

### ⚠️ 보안/운영 위험 요소
- **CORS 와일드카드(`*`)**: 현재 모든 오리진을 허용하고 있습니다. 운영 환경에서는 반드시 실제 프론트엔드 도메인만 허용하도록 명시해야 합니다.
- **CSRF 비활성화**: 만약 나중에 JWT를 쿠키에 담게 된다면, 이 설정을 다시 켜야 합니다.

## 🛠️ 2. DataInitializer.java (성 가동 준비단)

애플리케이션 시작 시 관리자 계정과 필수 데이터(테스트 쿠폰 등)를 자동으로 생성하는 '마중물' 역할을 합니다.

### 주요 해부점
- **멱등성(Idempotency)**: 이미 데이터가 있으면 만들지 않고, 없으면 만듭니다. 중복 생성을 방지합니다.
- **암호화 필수**: 관리자 계정 생성 시에도 반드시 `PasswordEncoder`를 사용하여 평문 저장을 방지합니다.
- **안전 잠금장치**: `try-catch`를 통해 시딩 작업 실패가 서버 구동 실패로 이어지는 것을 방지합니다.

### ⚠️ 운영 위험 요소
- **Hard-coded Credentials**: 관리자 비밀번호가 코드에 노출되어 있습니다. 배포 시에는 외부 환경 변수(`@Value`)로 분리해야 합니다.

---

## 🗝️ 3. PasswordEncoderConfig.java (암호 제작소)

- **기능**: 비밀번호를 사람이 읽을 수 없는 해시 형태(BCrypt)로 변환합니다.
- **왜 필요한가?**: 보안 사고 발생 시 사용자의 실제 비밀번호 노출을 원천 차단하기 위한 법적/기술적 필수 장치입니다.

---

## 💰 4. PortOneConfig.java (결제 API 연동)

- **apiSecret**: 결제 서버 인증을 위한 비밀 키입니다.
- **WebClient**: 성능과 동시성을 고려한 최신 방식의 HTTP 클라이언트를 사용합니다.

---

## 👮 5. MethodSecurityConfig.java (정교한 내부 보초)

메서드 단위에서 `@PreAuthorize` 권한 체크를 활성화합니다.

### ⚠️ 보안 위험 요소
- **@Profile("!dev") 주의**: 현재 개발 모드가 아닐 때만 보안이 작동합니다. 만약 운영 서버가 실수로 `dev` 프로필로 구동될 경우 모든 정밀 보안이 해제되는 대참사가 발생할 수 있습니다.

---

## 💡 실제 요청 내부 동작 순서

1. **CORS 확인**: 허용된 도메인의 요청인지 필터링.
2. **JWT 검문**: 헤더의 토큰 유효성 검사.
3. **권한 부여**: 토큰 기반 유저 롤(Role) 확정.
4. **URL 인가**: 요청한 경로가 해당 유저에게 허용된 길인지 확인.
5. **메서드 보안**: 최종 비즈니스 로직 진입 전 정교한 권한 체크.
6. **로직 실행**: 모든 관문을 통과하면 DB 조회/처리 수행.
